什麼是 vif？

vif 是一套以 Claude Code plugin 形式運作的 AI 驅動軟體開發流程框架。它將軟體開發從需求探索到交付收尾，拆解為明確的階段，每個階段由專屬的 skill 驅動，搭配 6 個專職 AI subagent 執行具體工作。

核心理念

  • Spec-First — 沒有核准的規格，就不寫程式
  • TDD 強制 — 沒有失敗的測試，就不寫產品程式碼
  • Impact Analysis — 每次變更先釐清新增 vs. 修改
  • 驗證 = 誠實 — 只宣稱有新鮮證據支持的結論
  • 文件即合約 — 設計文件是施工藍圖，不是擺設

三個 Hard Gate

Gate 1PRD → Spec確認問題定義與方向
Gate 2Spec → Develop確認技術範圍與設計文件
Gate 3Review → Close確認品質與規格符合度

兩種使用模式

模式一：完全自動化 Solo / 小團隊

AI 驅動所有階段，人類在 Gate 點審核決策。

技術先行：
/vif-arch + /vif-uiux → /vif-prd → /vif-spec → 設計 → 開發 → 驗證 → 審查 → 收尾

產品先行：
/vif-prd → /vif-arch + /vif-uiux → /vif-spec → ...

模式二：輔助自動化 企業團隊

各角色各自驅動對應的 skill。

架構師/vif-arch
PM / PD/vif-prd、/vif-bdd
設計師/vif-uiux、/vif-prototype
前端/vif-ui-spec、/vif-develop
後端/vif-api-spec、/vif-develop
彈性跳過機制：不是每個任務都需要走完全部階段。Bug 修復可跳過 PRD/BDD；設定檔變更可跳過 PRD/BDD/Spec；只有新功能開發需要完整流程。

Phase 0 探索與定義

釐清問題、建立技術與設計基礎。

/vif-prd 產品需求文件

做什麼：釐清「要解決什麼問題」和「為什麼值得解決」。vif-prd 是思考夥伴，不是表格填寫機。

流程

  1. 問題探索 — 從四個維度深入對話：問題本身、證據、預期成果、方案選項
  2. 撰寫 PRD — 依模板產出結構化文件
  3. Human 審核 — 核准後才能進入 Spec 階段

產出檔案

docs/prd-NNN.md

實際範例（Velora PRD-001 節錄）

# PRD-001: Velora GUI 桌面應用

## 1. 問題描述
使用者在完成語音轉錄後，需要花費大量時間進行逐字稿後處理...

## 4. 方案選項
| 方案 | 優點 | 缺點 |
|------|------|------|
| A. Tauri v2 + Svelte | 安裝包小、跨平台 | 生態較年輕 |
| B. Electron + React  | 生態成熟       | 安裝包大   |
| C. Python GUI (PyQt) | 直接呼叫 pipeline | UI 開發效率低 |

> 決定：A. Tauri v2 + Svelte

## 6. Spec 拆解建議
| Spec | 名稱 | 優先級 |
|------|------|--------|

PRD 的精髓在於「方案選項比較表」— 不只記錄選了什麼，更記錄為什麼不選其他方案。
/vif-arch 架構決策記錄（ADR）

做什麼：記錄架構決策的「為什麼」。寫給六個月後的自己看。

使用時機

  • 專案啟動時：技術棧選型、目錄結構、測試策略
  • 開發過程中：引入新技術、重構模組、效能優化

產出檔案

docs/architecture/adr-NNN-name.md

實際範例（Velora ADR-001 節錄）

# ADR-001: Pipeline 步驟進度回報策略

## 各引擎進度能力
| 引擎 | 進度機制 | 原生 callback |
|------|---------|--------------|
| faster-whisper | whisperx progress_callback | Yes |
| whisper-cpp | new_segment_callback | Yes（間接）|
| mlx-whisper | tqdm 輸出到 stderr | No |

## 決策
優先使用引擎原生 callback，無 callback 時
parse stderr 的 tqdm 輸出作為 fallback。
設定彈性：可在專案任何時間點觸發；會自動偵測是否已有 PRD 作為技術選型參考。
/vif-uiux UI/UX 設計基礎

做什麼：從產品意圖出發，建立有辨識度的設計系統（不是套用 AI 預設風格）。

Intent First 三問

  1. 誰在用這個？（具體場景）
  2. 他們在完成什麼？（一個動詞）
  3. 應該有什麼感覺？（溫度、密度、個性）

設計決策涵蓋

面向內容
色彩系統主色、輔色、功能色、中性色
字型字型家族與個性、大小級距
間距基礎單位、密度選擇
深度策略四選一：純邊框 / 淺陰影 / 層疊陰影 / 色差
元件規格按鈕、表格、表單、Modal
狀態表現default/hover/active/focus/disabled + loading/empty/error

品質檢驗（四項測試）

  • Swap test — 換掉字型/排版，看起來還有差異嗎？
  • Squint test — 模糊螢幕，還能看出層級嗎？
  • Signature test — 找 5 個體現設計方向的元素
  • Token test — CSS 變數名稱能聯想到產品世界嗎？

/vif-spec 技術規格與影響分析 Hard Gate

做什麼：整個流程的戰略核心 — 產出 spec.md（作戰計畫）和 progress.md（追蹤表）。

最重要的步驟：Impact Analysis — 掃描 PRD + 現有程式碼 + 設計文件，產出「新增 vs. 修改」的影響表格。這一步決定了整個 Spec 的品質。

/vif-api-spec API 規格 + DB Schema

做什麼：後端的完整施工藍圖 — 不只是 endpoint 清單，包含所有失敗情境。

每份 API Spec 包含

  • 方法、路徑、請求參數（含驗證規則）
  • 回應（成功 + 每種錯誤）
  • 完整業務邏輯流程
  • 邊界條件
  • 錯誤映射表（所有可能的失敗情境）
  • 呼叫範例（成功 + 錯誤）

實際範例（Velora list_cached_models）

## 錯誤處理
| 操作 | 錯誤情境 | Error Code | 使用者看到 |
|------|---------|------------|-----------|
| invoke | 無法啟動 Python | PYTHON_SPAWN_FAILED | 無法啟動：{detail} |
| invoke | 逾時（15 秒） | PYTHON_TIMEOUT | 模型管理操作逾時 |
| invoke | 回傳非 JSON | MODEL_LIST_FAILED | 無法載入：{detail} |

## 邊界條件
- HuggingFace cache 不存在 → 回傳空陣列
- 大量非 Velora 模型 → 只回傳已知 repo_id
- Python stderr 有 warning → 忽略，只讀 stdout

設定彈性：非 REST API 專案（如 Tauri IPC）可在 CLAUDE.md 自訂規範，例如取消 openapi.yaml、合併錯誤處理表格。

Phase 2 實作

以嚴格的 TDD 紀律執行 spec.md 的任務清單。

/vif-develop TDD 開發

進入條件

  • spec.md 已核准
  • progress.md 的設計文件表全部完成（無「待撰寫」）
  • 所有設計文件已通過自審

核心循環：RED → GREEN → REFACTOR

🔴
RED
test-writer agent
撰寫失敗的測試
確認失敗是合法的
🟢
GREEN
implementer agent
寫最少的程式碼
讓測試通過
🔵
REFACTOR
消除重複
改善命名
保持測試通過

Implementer 回報狀態

狀態意義後續動作
DONE完成，測試通過繼續下一個任務
DONE_WITH_CONCERNS完成但有疑慮記錄疑慮，繼續
NEEDS_CONTEXT需要更多資訊回饋迴圈（最多 1 次）
BLOCKED無法繼續升級給 Human

實際的 progress.md 追蹤（Velora Spec-006）

- [x] Task 1: model_registry.py — 共用模型映射
  - RED: 19 tests，失敗原因：ModuleNotFoundError ✓
  - GREEN: 建立純資料映射模組 ✓
  - Test: Unit ✓（22 passed + 全部 81 passed）
- [x] Task 2: model_manager.py — 模型列舉
  - RED: 8 tests，失敗原因：ModuleNotFoundError ✓
  - GREEN: 實作 list_cached_models() ✓
  - Test: Unit ✓（8 passed）
- [x] Task 3: model_manager.py — 模型刪除
  - RED: 9 tests，失敗原因：ModuleNotFoundError ✓
  - GREEN: 實作 delete_cached_model() ✓
  - Test: Unit ✓（13 passed + 全部 102 passed）

完成後的 De-Sloppify Pass

移除 console.log / debugger、註解掉的程式碼、清理 import、檢查不必要的 any、移除 TODO hack。

設定彈性：測試策略可在 CLAUDE.md 預設或每次選擇（Unit / Integration / E2E）。獨立任務可並行執行。TDD 例外需 Human 核准。

Phase 3 驗證

/vif-verify 自動化驗證 Pipeline

原則：沒有新鮮的驗證證據，就不能宣稱完成。

驗證階段（全部執行，即使前面的失敗了）

#階段內容
1Build建構專案
2Type Check型別檢查
3Lint程式碼風格
4Test Suite完整測試
5Diff Review變更分析
6Dependency Audit依賴安全
7Security ReviewOWASP Top 10 靜態分析

結果分類

PASS — 無問題   WARN — 需評估   FAIL — 必須修復

WARN 處理：每個 WARN 必須被評估 — 修復或記錄接受原因。沒有記錄原因的 WARN = 未評估 = 不算通過。

驗證報告範例

## 結果摘要
| 階段 | 狀態 | 說明 |
|------|------|------|
| Build | ✅ | 成功 |
| Lint | ⚠️ | 4 unused imports |
| Test | ✅ | 102 passed |
| Security | ✅ | 0 critical, 0 major |

## WARN 評估
- Lint: 4 unused imports → auto-fixed ✓
- Audit: pre-existing 上游依賴 → 接受 ✓

## 判定：READY for Code Review

失敗時回到 develop 修復，必須重跑完整 pipeline（不能只跑失敗的階段）。

Phase 4 審查

/vif-review 兩階段程式碼審查 Hard Gate

原則：不信任報告，自己驗證。

Stage 1：規格符合度（不通過不進 Stage 2）

  • 驗收條件核對 — 每個 AC 都有對應的實作 + 測試 + 實際行為
  • 設計文件一致性 — API 實作 vs. api-spec、UI 實作 vs. ui-spec
  • 範圍確認 — 變更的檔案是否在 spec 計畫內

Stage 2：程式碼品質

  • 架構（模組切分、依賴方向）
  • 可讀性（命名、結構）
  • 測試品質（行為驗證、邊界案例）

Issue 嚴重度

等級含義處理
Critical違反規格或破壞功能必須修復
Major影響維護性/效能應該修復
Minor規格外的改善建議可選

審查通過後產出 Manual Testing Checklist，Human 完成手動測試並最終核准。

修復迴圈：發現問題 → develop 修復 → 重跑 verify → 重跑 review，最多 3 次。

Phase 5 收尾

/vif-close 開發完成與文件同步

工作內容

  1. 設計文件同步 — 如果實作偏離設計，更新 api-spec / ui-spec / schema，讓下一個讀者不被誤導
  2. 完成檢查清單：
    • 所有 scenario 已實作 + 測試通過
    • 測試覆蓋率 ≥ 80%（核心路徑 ≥ 90%）
    • 所有 Critical / Major issue 已解決
    • Security review 通過
    • progress.md 所有階段打勾
    • specs-overview.md 狀態更新為 done
  3. 版本控制 — 所有變更 commit，必要時打 git tag

設定與彈性

在專案的 .claude/CLAUDE.md 中設定 vif。

基本設定範例

## AI-Driven Development Flow

本專案採用 vif。

### 技術棧
- 語言：Python
- 測試：pytest
- 建構：uv

### 專案指令
- Build: `uv build`
- Test: `uv run pytest`
- Lint: `uv run ruff check`

### Human 介入點
| Gate | Human 行為 | 說明 |
|------|-----------|------|
| PRD → Spec | Approve PRD | 確認問題定義 |
