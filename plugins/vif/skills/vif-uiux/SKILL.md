# UI/UX — 設計基礎

建立專案的 UI/UX 設計基礎，定義視覺風格、色彩、字型、元件規範。確保所有頁面有一致的設計語言。

## Intent First

在碰任何視覺元素之前，先具體回答三個問題：

1. **誰在用這個產品？** 不是「使用者」。是哪個人？他打開這個畫面時在哪裡？腦子裡在想什麼？
2. **他要完成什麼？** 一個動詞。批改作業。找到壞掉的部署。核准付款。
3. **這應該感覺像什麼？** 「乾淨現代」什麼都沒說。溫暖像筆記本？冷冽像終端機？密集像交易大廳？

如果回答不出具體的，停下來問 Human。不要用預設值。

> **每個選擇都必須是一個選擇。** 對每個決定解釋 WHY — 佈局、色溫、字型、間距、層次。
> 如果你的答案是「這很常見」或「這行得通」，你就是在用預設值。

## Stance

**設計要從產品領域出發，不是套模板。**

- 色彩要感覺「屬於」這個產品，不是「被套用」到這個產品
- 避免 AI 預設風格：Inter 字型、紫色漸層、統一圓角、置中佈局
- Typography 是設計，不是容器。一個烘焙工具和交易終端都需要「可讀的字型」— 但溫暖的字型不是精確的字型
- Navigation 是產品，不是鷹架。一個浮在空白中的頁面是元件 demo，不是軟體
- Token 名稱是設計語言。`--ink` 和 `--parchment` 喚起一個世界，`--gray-700` 和 `--surface-2` 喚起一個模板

## 使用情境

### A. 專案啟動（首次）

與 Human 討論並建立 UI/UX 設計基礎。

**Step 1: 產品領域探索（不可跳過）**

在提出任何方向之前，必須產出以下四項：

1. **領域** — 這個產品世界裡的概念、隱喻、詞彙（至少 5 個）
2. **色彩世界** — 這個領域自然存在什麼顏色？如果這個產品是一個實體空間，你會看到什麼？（至少 5 個）
3. **標誌性元素** — 一個只有這個產品才會有的元素
4. **刻意不用的預設** — 3 個這類介面最常見的預設選擇，以及你用什麼取代、為什麼

```
"領域：[5+ 概念]
色彩世界：[5+ 從領域推導的顏色]
標誌性元素：[這個產品獨有的元素]
拒絕的預設：[預設 1] → [替代], [預設 2] → [替代], [預設 3] → [替代]

設計方向：[連結以上四項的方向描述]"

→ 詢問 Human：「這個方向感覺對嗎？」
```

**Step 2: 設計決策**

與 Human 逐項討論並記錄（使用 `references/ui-guideline-template.md`）。

每一項都要說明 **WHY**，不只填值：

1. **色彩系統** — 從產品領域推導，主色為什麼選這個？
2. **字型系統** — 選擇有個性的字型，為什麼這個字型適合這個產品？
3. **間距與佈局** — 定義基準單位，密集還是寬鬆？為什麼？
4. **深度策略** — 選一種並堅持：純邊框 / 淡陰影 / 層疊陰影 / 表面色差
5. **元件規範** — 按鈕、表格、表單、Modal 的風格
6. **回饋機制** — Toast、確認彈窗、錯誤訊息
7. **狀態表示** — 各狀態的顏色和徽章

**Step 3: 品質測試**

定義完成後，必須通過四個測試：

| 測試 | 問題 | 不通過的徵兆 |
|------|------|------------|
| **Swap test** | 把字型/佈局換成常見選擇，感覺有明顯不同嗎？ | 換了也沒差 → 選擇太通用 |
| **Squint test** | 把畫面模糊，還能看出層次嗎？ | 模糊後分不清區塊 → 對比不足 |
| **Signature test** | 能指出 5 個體現設計方向的元素嗎？ | 找不到 → 設計缺乏個性 |
| **Token test** | CSS 變數名讀出來，像在說這個產品的世界嗎？ | 全是 `--gray-100` → 缺乏語義 |

任何一項不通過 → 迭代修正後再呈現。

**Step 4: 確認與記錄**

- 呈現完整的設計基礎給 Human 確認
- 存檔至 `guideline/ui/ui-guideline.md`
- **commit**

### B. 設計調整（持續）

開發過程中需要調整設計決策時：
- 更新 `guideline/ui/ui-guideline.md`
- 說明為什麼調整（記錄在版本歷程中）
- **commit**

## Craft Foundations

以下原則不管設計方向如何都適用，是品質底線。

### 表面層級

表面是堆疊的。下拉選單浮在卡片上，卡片浮在頁面上。建立編號系統：

```
Level 0: 基礎背景（app 畫布）
Level 1: 卡片、面板（與基礎同平面）
Level 2: 下拉選單、Popover（浮在上方）
Level 3: 巢狀下拉、堆疊覆蓋（罕用）
```

**微妙原則**：層級之間的差異要細微 — 幾個百分點的明度，不是戲劇性的跳躍。你幾乎看不到差異，但你感覺得到。

### Token 架構

每個顏色都要追溯到基本語意：
- **Foreground** — 文字色（4 級層次：primary / secondary / tertiary / muted）
- **Background** — 表面色（base / elevated / overlay）
- **Border** — 邊框色（default / subtle / strong）
- **Brand** — 品牌主色
- **Semantic** — 功能色（success / warning / danger / info）

不要出現隨機的 hex 值。每個值都要能解釋是哪個 token。

### 深度策略

選一種，然後全專案堅持：

| 策略 | 感覺 | 適合 |
|------|------|------|
| **純邊框** | 乾淨、技術、密集 | 開發者工具、工具型產品 |
| **淡陰影** | 柔和、平易近人 | 一般 SaaS |
| **層疊陰影** | 豐富、高級、立體 | 需要質感的產品 |
| **表面色差** | 背景色漸變建立層次，無陰影 | 極簡風格 |

### 互動狀態

每個可互動元素都必須定義五種狀態：

| 狀態 | 說明 |
|------|------|
| default | 預設外觀 |
| hover | 滑鼠移入 |
| active | 點擊中 |
| focus | 鍵盤聚焦（無障礙必要） |
| disabled | 不可操作 |

資料元素額外需要：loading / empty / error 三種狀態。

遺漏狀態 = 感覺壞掉了。

## 開發時的思考清單

每次寫 UI 元件時，先陳述：

```
Intent: [誰、在做什麼、應該感覺像什麼]
Palette: [從探索中來的顏色 — 為什麼適合]
Depth: [邊框 / 陰影 / 層疊 — 為什麼]
Surfaces: [層級系統 — 為什麼這個色溫]
Typography: [字型 — 為什麼適合這個意圖]
Spacing: [基準單位]
```

如果任何一項解釋不出 WHY，你就是在用預設值。停下來想。

## 絕對不要

- 通用 AI 風格（Inter/Roboto、紫色漸層、白底）
- 可預測的佈局和元件模式
- 缺乏產品特色的複製貼上設計
- 生硬的邊框、戲劇性的表面跳躍
- 混用多種深度策略
- 不同色相用在不同表面 — 同色相，只調明度
- 同樣的產出出現第二次 — 每次都要有變化
